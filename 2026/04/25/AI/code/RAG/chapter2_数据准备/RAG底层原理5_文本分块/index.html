RAG底层原理5:文本分块

发表于 2026-04-25 更新于 2026-06-07 分类于 AI ， 开发 ， RAG 阅读次数：

前言：

文本分块（Text Chunking）是构建 RAG 流程的关键步骤。它的原理是将加载后的长篇文档，切分成更小、更易于处理的单元。这些被切分出的文本块，是后续向量检索和模型处理的基本单位。

它的角色：它是数据加载之后的二次加工，属于模块二：数据清洗与分块策略 (Data Processing) 阶段的第二步

它要解决的问题（容量与精度）：

  1. 容量受限：大模型（LLM）和向量提取模型（Embedding）都有严格的字数限制（Token 限制）。如果你把一整本 10 万字的书直接扔进去，模型会被直接“撑死”报错。

  2. 检索稀释：如果你不切块，把一整篇文章当成一个目标去检索，里面的关键信息就全被稀释了。系统很难判断这篇文章到底是在讲历史、讲科技还是讲菜谱。

它的动作：分块器（Splitter）会按照你设定的规则（比如每 500 字一刀，且保留 50 字的上下文重叠），把加载进来的超长文本切成一小段、一小段的“文本块”（Chunk）。

分块原则：

首先块不能太大，也不能太小
一：为什么块不能“太大”？

如果为了省事，把一整页纸甚至一整章切成一个大块（比如一次性切 3000 字），会引发致命的灾难：

  1. 撑爆模型的“胃口”（Token 限制）： 把文本变成向量的 Embedding 模型（比如 bge-m3 或 OpenAI 的 text-embedding-3）都有严格的输入长度限制（通常是 512 到 8192 个 Token）。如果文本块超长，超出的部分会被模型直接无情截断，信息当场丢失。

  2. “语义稀释”导致检索失灵（大海捞针）： 假设一个大文本块里同时讲了“苹果公司的财报”、“苹果手机的销量”和“种苹果树的技术”。当用户搜索“如何种水果”时，这个块虽然包含了正确答案，但由于“噪音”太多，它的整体向量语义会被拉偏。在计算相似度时，它反而竞争不过那些短小精悍、只讲种树的文本。

  3. 增加大模型的“幻觉”和成本： 就算检索到了这个巨大的块并喂给大模型，大模型要在茫茫字海中找出一句关键的话，不仅消耗极其昂贵的 Token 费用，还极其容易产生“注意力偏移”（Attention 丢失），从而胡言乱语。

二：为什么块不能“太小”？

既然太大不好，那我一句话（甚至几个词）切一块（比如 20 个字）行不行？同样也是灾难：

  1. 致命的“上下文丢失”（指代不明）： 如果切块太小，很容易把一句话劈成两半。比如检索库里有一块的内容仅仅是：_“它在 2024 年第一季度的净利润暴涨了 50%。”_ 大模型拿到这句话会彻底抓狂——“它”是谁？！由于失去了上一段的上下文，这块数据成了毫无价值的“废料”。

  2. 检索碎片化与错位： 太小的块语义极其单一。用户提问一个稍微复杂的问题（例如需要结合起因和结果的复合问题），系统可能只能搜到零碎的片段，无法为 LLM 提供支撑完整逻辑链的素材。

文本分块的“黄金三大原则”

基于上述的两个极端，架构师在设计 RAG 分块策略时，必须死守以下三大原则：

原则一：语义完整性第一

永远不要生硬地“按字数一刀切”。优秀的切块策略必须尊重人类语言的自然结构，优先按段落切分，其次按句子切分，最后才考虑按字符切分。要尽量保证每一个切出来的块，人类单独抽出来阅读时，也能看懂它在表达什么。

原则二：保持上下文重叠

为了防止“切断骨头连着筋”的信息被割裂，相邻的两个文本块之间必须要有重叠区域（Chunk Overlap）。 _就像拍全景照片一样，两张照片之间必须有重合部分，才能拼接出完整的风景。上一块的结尾 50 个字，必须是下一块的开头 50 个字，以此来强行保留上下文。_

原则三：因地制宜，任务导向

没有绝对完美的块大小，只有最适合当前任务的大小：

  • 如果是做百科问答、名词解释： 用户提问很精准，适合较小的块（如 200-500 Token），提高检索命中率。

  • 如果是做总结摘要、长逻辑推理： LLM 需要宏观信息，适合较大的块（如 1000-2000 Token），保证上下文的充沛。

这里先补充一个chunk_size的知识，chunk_size底层默认情况下，chunk_size 的单位是 字符数（Characters），不是字节，因为它底层默认使用的是 Python 自带的 len() 函数。对于 Python 字符串来说，len("你好") 算作 2 个字符，而不是 UTF-8 编码下的 6 个字节。

💡 进阶避坑： 在工业级 RAG 项目中，为了精准对齐大模型的输入限制，架构师通常会把单位换成 Token。做法是把默认的 len 换成分词器（比如 OpenAI 的 tiktoken），这样 chunk_size=500 就代表严格的 500 个 Token。

其实这里的固定大小，并不是真的我们设置chunk_size = 500，他就严格按照500字符切一刀，那样可能我们的语义会被切的七零八落

这种方法的工作原理其实分为两个主要阶段：

  1. 按段落分割 (拆碎)：它首先认准你设定的 separator（默认是 \n\n），用正则表达式把整篇文章劈成一个个独立的自然段。

  2. 智能合并 (拼装)：调用底层的 _merge_splits 方法，把刚切好的段落一段段往里拼。只要累积长度没超过 chunk_size，它就继续拼。一旦加上下一段会超标，它就把当前拼好的内容打包成一个新 Chunk，并通过 chunk_overlap 留一点尾巴给下一个 Chunk 保持连贯。

基于 _merge_splits 源码逻辑，你会发现它有三个合并原则

  • 段落完整性 > 强制字数：它绝对优先保证段落不被破坏。只有当“再塞一个完整的新段落就会撑爆大小”时，才会结束当前块。

  • 超长段落免疫（警告但不切）：如果你的文档里有一个长达 2000 字的超级大段落（中间没有双换行），而你设置的 chunk_size 是 200。它不会把这个段落从中间生生截断，系统会发出一条超长的警告（Warning），但依然会将这 2000 字作为一整个完好的 Chunk 保留下来。

  • 适用场景：实现简单、计算开销极小。非常适合处理日志文件、代码库、数据预处理或者格式极其规整（段落均匀）的文本。但在处理普通的长篇无排版文章时，可能不太灵活。

递归字符文本分割器 (RecursiveCharacterTextSplitter)

这是 RAG 系统中最通用、最推荐、也是最聪明的基础切分器。第一把刀遇到超长段落会“直接摆烂报警”，而这第二把刀则是专门为了对付超长段落而生的：遇到长段落，它会换更细的刀继续切。

具体逻辑：

  1. 找分隔符：从你配置的 separators 列表中，从左到右找第一个在文本中存在的分隔符（比如先找 \n\n）。

  2. 切分与暂存：把文本切碎。然后遍历这些碎片：

    • 如果碎片 <= chunk_size：说明切得很好，把它暂时放进一个叫 _good_splits（合格暂存池）的地方，等待后续合并，而不是立刻输出。

    • 如果碎片 > chunk_size：遇到“硬骨头”了！此时触发两连击：

      • 先强制把 _good_splits 里之前暂存的合格碎片打包成一个 Chunk 输出。

      • 然后，更换更细的刀（拿列表中剩下的分隔符，如句号 。），对这个超长硬骨头进行递归调用 (_split_text) 继续往下切

  3. 递归的底线：当所有的分隔符全试完了（new_separators 为空，连 "" 单字符切分都用过了），如果这块文本依然超长（极其罕见），它才会停止递归，被迫将其作为超长块保留。

语义分块

传统的固定长度和递归分块，虽然好用，但本质上还是在“靠数标点符号”来盲猜语义。而 SemanticChunker 则是真正让 AI 去“读懂”文章，在话题发生转移（语义跳跃）的地方精准切上一刀。LangChain 提供了 langchain_experimental.text_splitter.SemanticChunker 来实现这一功能。

核心算法流程 (五步)

  1. 句子分割：先按句号、问号等把文章拆成一句句话。

  2. 上下文感知嵌入[天才设计]：

    • 传统的做法是对每一句孤立的话算 Embedding。

    • 这里用到了 buffer_size 参数（滑动窗口）。它会把当前句子和它前后的句子打包在一起计算 Embedding 向量。这样算出来的向量，完美融合了上下文的语境，防止了单句歧义。

  3. 计算语义距离：计算相邻两个句子向量的余弦距离。距离越大，说明这两句话聊的内容跳跃越大（比如上一句聊李白，下一句突然聊火箭）。

  4. 识别断点：利用统计学方法算出一个“阈值”（具体看下文）。只要两句话的距离超过这个阈值，这里就是一个“断点”。

  5. 合并成块：沿着识别出来的断点手起刀落，把断点中间的所有句子融合成一个高度语义连贯的 Chunk。

四大“断点阈值”计算方法

如何定义“多大的跳跃才算跳跃”？它内置了四种统计学方法：

  1. percentile (百分位法，默认推荐)：把所有距离排个序，取前 5% 最陡峭的跳跃点作为切分点（amount=95）。

  2. standard_deviation (标准差法)：超过“平均距离 + N倍标准差”的算作异常跳跃点。

  3. interquartile (四分位距法)：统计学中的 IQR，超过 Q3 + N*IQR 视为断点。

  4. gradient (梯度法)：计算距离的“变化率”（二阶导数思想），对那些行文极其枯燥连贯的文档（如法律、医疗说明书）有奇效，能敏锐捕捉微小的语义拐点。

  3. 精准捕捉“话题转移点”：仔细看它的切分刀落在了哪里——它在文档从“基础履历/身高体重等元数据”过渡到“具体的游戏实战机制（技能与定位）”的转折处，精准地切下了一刀。它用 Embedding 向量算出来了：前文在聊“人设”，后文在聊“打法”，这里就是语义的跳跃点

结构化分块器

如果我们要处理的文档本身就是结构极其严谨的 Markdown、HTML 或 LaTeX，我们完全没必要用算法去“盲猜”断点。结构化分块的核心逻辑是：“既然作者已经用 # 和 ## 把章节画好了，我们就直接按作者的骨架来切

核心逻辑：元数据注入

普通的切分器切出来的块，就像是一张张散落的卡片，大模型拿到一张卡片时，完全不知道它属于哪一章。而 MarkdownHeaderTextSplitter 是一台会给卡片打 “GPS 定位标签” 的机器。

  • 内容聚合：它会把 # 到下一个 # 之间的所有内容打包。

  • 地址标记：比如一句话位于“第三章”下的“3.2节”。切出来的块会自动带上这样一个隐形标签：{"Header 1": "第三章", "Header 2": "3.2节"}。

  • RAG 实战优势：大模型收到这段话时，不仅看到了文字，还能通过元数据立刻明白它的宏观背景。这就彻底解决了“只见树木，不见森林”的上下文丢失问题。

局限性：

如果一篇文章的“3.2节”长达一万字，单纯按标题切分依然会撑爆大模型。

为了解决这个问题，MarkdownHeaderTextSplitter 可以与其它分块器（如 RecursiveCharacterTextSplitter）组合使用。具体流程是：

  • 第一步，使用 MarkdownHeaderTextSplitter 将文档按标题分割成若干个大的、带有元数据的逻辑块。
  • 第二步，对这些逻辑块再应用 RecursiveCharacterTextSplitter，将其进一步切分为符合 chunk_size 要求的小块。由于这个过程是在第一步之后进行的，所有最终生成的小块都会继承来自第一步的标题元数据。

RAG应用优势: 这种两阶段的分块方法，既保留了文档的宏观逻辑结构（通过元数据），又确保了每个块的大小适中，是处理结构化文档进行RAG的理想方案。
